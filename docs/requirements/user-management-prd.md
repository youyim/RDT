# 用户管理 (User Management) 需求文档 (PRD)

## 1. 概述

### 1.1 背景

作为研发全流程管理系统配置中心的一部分，我们需要一个全局的用户管理功能，以便管理员能够高效地维护系统用户数据。

### 1.2 目标

构建一个美观（玻璃态设计）、易用且功能纯粹的 CRUD 用户管理控制台，支持用户信息的增删改查及账号状态管理。

### 1.3 范围

**包含**:

- 全局用户列表展示（姓名/头像、企业邮箱、账号状态）
- 用户搜索功能
- 新增用户功能
- 编辑用户信息
- 重置用户密码
- 归档/删除用户

**不包含**:

- 复杂的权限/角色分配（Phase 2）
- 组织架构管理（Phase 2）
- **明确**: MVP 版本不维护“角色”和“部门”字段，仅基础用户信息。

### 2.2 用户故事

#### US-001: 查看用户列表

- **As a** 系统管理员
- **I want** 在一个设计精美的表格中查看所有系统用户
- **So that** 我能掌握当前系统的用户概况

**验收标准**:
| AC | Given | When | Then | ✓ |
|----|-------|------|------|---|
| 1 | 管理员进入用户管理页面 | 页面加载完成 | 展示用户列表，包含：头像、姓名、企业邮箱、状态 | [ ] |
| 2 | 列表样式 | 页面渲染 | 表格采用大面积玻璃态设计，视觉通透 | [ ] |

#### US-002: 搜索用户

- **As a** 系统管理员
- **I want** 通过关键词搜索用户
- **So that** 我能快速找到特定的用户进行管理

**验收标准**:
| AC | Given | When | Then | ✓ |
|----|-------|------|------|---|
| 1 | 用户列表中有数据 | 在顶部搜索框输入姓名或邮箱 | 列表实时或回车后过滤显示匹配结果 | [ ] |

#### US-003: 新增用户

- **As a** 系统管理员
- **I want** 添加新的用户账号
- **So that** 新成员可以登录系统

**验收标准**:
| AC | Given | When | Then | ✓ |
|----|-------|------|------|---|
| 1 | 在用户管理页面 | 点击“新增用户”按钮 | 弹出新增用户模态框/抽屉 | [ ] |
| 2 | 填写表单 | 输入必要信息（姓名、邮箱）并保存 | 生成新用户记录（不包含角色/部门），列表自动刷新 | [ ] |

#### US-004: 管理用户操作

- **As a** 系统管理员
- **I want** 对单个用户进行编辑、重置密码或删除
- **So that** 我能维护用户信息的准确性和安全性

**验收标准**:
| AC | Given | When | Then | ✓ |
|----|-------|------|------|---|
| 1 | 在用户列表某一行 | 鼠标悬停或点击操作列 | 显示“编辑”、“重置密码”、“归档/删除”按钮 | [ ] |
| 2 | 编辑用户 | 点击“编辑” | 弹出编辑框，回显当前信息，保存后更新 | [ ] |
| 3 | 重置密码 | 点击“重置密码” | 弹出确认提示“是否发送重置密码邮件？”，确认后前端提示发送成功（后端保留接口） | [ ] |
| 4 | 删除用户 | 点击“归档/删除” | 弹出二次确认框，确认后进行软删除（状态变更为已归档/删除） | [ ] |

## 3. 非功能性需求

| 类型     | 要求                                     | 验收方式   |
| -------- | ---------------------------------------- | ---------- |
| 交互体验 | 列表滑动流畅，玻璃态效果不影响文字可读性 | 视觉走查   |
| 响应速度 | 列表加载 < 1s (支持分页)                 | 性能测试   |
| 适配性   | 适应主流桌面分辨率                       | 兼容性测试 |
| 安全性   | 重置密码接口需鉴权                       | 代码评审   |

---

## 4. 约束与假设

### 假设

- 假设后端已准备好相应的 API 接口（或将在 Phase 2a 中同步设计）。
- 假设“归档/删除”操作主要为软删除（归档），数据库中保留记录。
- **重置密码**：MVP 阶段仅需前端展示发送动作，后端提供预留接口（Mock 或空实现），暂不真实发送邮件。
- **用户模型**：不包含角色（Role）和部门（Department）关联。
